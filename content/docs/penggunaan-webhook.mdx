Webhook digunakan agar sistem partner bisa menerima event dari Socialchat secara otomatis. Dengan webhook, aplikasi kamu tidak perlu polling API terus-menerus untuk mengetahui pesan baru, perubahan percakapan, atau event integrasi lainnya.

## Apa yang bisa dilakukan webhook?

Webhook dapat digunakan untuk beberapa kebutuhan integrasi, misalnya:

- Menerima pesan masuk dari pelanggan.
- Menerima update percakapan atau kontak.
- Meneruskan pesan ke CRM internal.
- Menjalankan automation ketika ada pesan baru.
- Membuat tiket support dari conversation Socialchat.
- Menyimpan arsip pesan ke database partner.
- Menghubungkan Socialchat dengan sistem order, loyalty, chatbot, atau helpdesk.

## Prasyarat

Sebelum mengatur webhook, pastikan kamu sudah memiliki:

- API key Socialchat Partner API.
- URL webhook publik yang bisa menerima request `POST`.
- Endpoint webhook menggunakan HTTPS.
- Server webhook membalas response `2xx` dengan cepat.

Contoh URL webhook:

```txt
https://api.example.com/socialchat/webhook
```

## 1. Cek konfigurasi webhook saat ini

Referensi API: [GET Partner](/docs/openapi/partner-api/PartnerController_getPartner)

Gunakan endpoint Partner untuk melihat konfigurasi webhook yang sedang aktif.

```bash
curl -X GET "https://api.socialchat.id/partner/" \
  -H "Authorization: Bearer ***" \
  -H "Content-Type: application/json"
```

Contoh response:

```json
{
  "webhookUrl": "https://api.example.com/socialchat/webhook",
  "webhookSecondaryUrls": [
    "https://api.example.com/socialchat/webhook-secondary"
  ]
}
```

Field penting:

- `webhookUrl`: URL utama penerima event Socialchat.
- `webhookSecondaryUrls`: daftar URL tambahan jika event perlu dikirim ke lebih dari satu endpoint.

## 2. Update URL webhook

Referensi API: [PUT Partner](/docs/openapi/partner-api/PartnerController_updatePartner)

Gunakan endpoint update Partner untuk menyimpan URL webhook.

```bash
curl -X PUT "https://api.socialchat.id/partner/" \
  -H "Authorization: Bearer ***" \
  -H "Content-Type: application/json" \
  -d '{
    "webhookUrl": "https://api.example.com/socialchat/webhook",
    "webhookSecondaryUrls": [
      "https://api.example.com/socialchat/webhook-secondary"
    ]
  }'
```

Jika hanya memakai satu endpoint webhook, isi `webhookUrl` saja dan gunakan `webhookSecondaryUrls` sebagai array kosong atau tidak dikirim jika tidak dibutuhkan.

## 3. Siapkan endpoint penerima webhook

Endpoint webhook di server partner harus menerima request `POST` dengan body JSON.

Contoh sederhana menggunakan Express:

```js
const express = require('express')
const app = express()

app.use(express.json({ limit: '5mb' }))

app.post('/socialchat/webhook', async (req, res) => {
  const payload = req.body

  console.log('Webhook Socialchat:', JSON.stringify(payload, null, 2))

  // Proses event secara async jika butuh pekerjaan berat.
  // Simpan payload ke queue/database, lalu balas 200 secepat mungkin.

  return res.status(200).json({ received: true })
})

app.listen(3000, () => console.log('Webhook server running'))
```

Rekomendasi:

- Balas `200 OK` secepat mungkin.
- Jangan menjalankan proses berat sebelum response.
- Simpan payload ke queue jika perlu integrasi lanjutan.
- Pastikan endpoint idempotent agar aman jika event dikirim ulang.

## 4. Isi webhook yang diterima

Payload webhook mengacu pada data yang dikirim ke queue webhook Socialchat. Secara internal, event dikirim dengan format `type` dan `data`, lalu body yang diterima endpoint webhook menjadi:

```json
{
  "type": "message",
  "...data": "..."
}
```

Artinya field `type` berada di level paling atas, lalu isi `data` digabung ke payload utama. Bentuk data dapat berbeda sesuai jenis webhook.

### Type webhook yang umum

- `message`: pesan masuk dari channel seperti WhatsApp API, WhatsApp Unofficial, Telegram, Instagram, Messenger, Webchat, atau Linkchat.
- `message-status`: status atau hasil pengiriman pesan.
- `contact`: event kontak, misalnya kontak baru atau kontak yang diproses oleh integrasi.

## Field penting pada payload

- `type`: jenis webhook yang diterima, misalnya `message`, `message-status`, atau `contact`.
- `conversation._id`: ID percakapan di Socialchat.
- `conversation.senderId`: ID atau nomor pengirim.
- `conversation.senderName`: nama kontak/pengirim.
- `message._id`: ID pesan di Socialchat.
- `message.messageId`: ID pesan dari channel asal jika tersedia.
- `message.type`: tipe pesan, misalnya `text`, `media`, `location`, `contact`, `product`, atau `polling`.
- `message.text`: isi pesan teks.
- `message.media`: detail file jika pesan berupa media.
- `message.channelBy._id`: ID channel Socialchat.
- `message.channelBy.channelType`: tipe channel, misalnya `whatsapp-official`, `whatsapp-unofficial`, `telegram`, `instagram`, `messenger`, `webchat`, atau `linkchat`.

## Best practice

- Validasi bahwa request datang ke endpoint yang benar.
- Simpan raw payload untuk kebutuhan debugging.
- Gunakan queue/job worker untuk proses berat seperti kirim ke CRM atau upload file tambahan.
- Balas `2xx` secepat mungkin agar Socialchat menganggap webhook berhasil diterima.
- Hindari response lambat atau timeout karena dapat menyebabkan retry atau event terlihat gagal.
- Jangan menaruh secret/API key di URL webhook.

## Troubleshooting

- Jika webhook tidak diterima, cek ulang `webhookUrl` di Partner API.
- Pastikan URL webhook publik, HTTPS, dan tidak diblokir firewall.
- Pastikan server mengizinkan method `POST` dan header `Content-Type: application/json`.
- Jika payload media tidak bisa diakses, cek apakah URL media masih valid dan dapat diakses dari server partner.
- Jika event dobel, buat proses idempotent berdasarkan `message._id` atau `message.messageId`.
